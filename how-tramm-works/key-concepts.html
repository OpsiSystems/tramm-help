Key Concepts

Tramm is three products working as one system — from the moment an order is created to the moment it is delivered and confirmed. This page explains how everything fits together.

The Big Picture

Everything in Tramm flows through a clear data model. A client request becomes an Order. Orders are broken into Jobs. Jobs are grouped into Trips. Trips have Stops. Work happens at Stops through Activities. Understanding this hierarchy makes the rest of the system straightforward.

Everything starts with reference data. The sites, vehicles, products, routes, and operating hours that appear throughout this guide all need to be set up before planning can begin. They are configured and maintained in Data Manager, the reference data hub within Tramm TMS. Whenever this guide refers to a site's entry duration, a vehicle's capacity, or a product's loading rate, that setup lives in Data Manager.

Three Products, One System

Most Tramm customers use all three products together. They share the same data — a trip planned in Tramm TMS automatically appears for the driver in Tramm Mobile and is visible in real time on Fleetvision.

Orders

Order

What the client asked for

Used in
Order Manager

An order is a client's request for a delivery, collection, or service at a specific site. It captures what needs to happen — not how it will be done. An order has no knowledge of vehicles, routes, or scheduling. That comes later when jobs are created.

Orders are optional in Tramm. Some operations work entirely with jobs and never create orders — for example, when trips are built directly without a formal order intake process. Where orders are used, they are the entry point that drives job creation.

Key points

  • Every order specifies a site (where), a time window (when), and the service type: delivery (goods to a site), collection (goods from a site), or service (work at a site with no goods movement).
  • For deliveries and collections, an auxiliary site is also specified — typically the depot where goods will be loaded onto or returned from the vehicle. The two sites are labelled differently depending on job type: for a delivery, the labels are typically Loading site (depot) and Delivery site (client); for a collection, Collection site (client) and Offload site (depot).
  • Orders can be entered manually in Order Manager, or imported automatically from an external system such as a warehouse management system or customer portal.
  • If an order is too large for a single vehicle, it is split into multiple jobs when the planner reviews it. A large order does not become multiple orders — it stays as one order with multiple jobs.
  • Orders are not sent to the scheduler directly — jobs are. An order drives the process; jobs do the work.
States New → Partially Planned → Planned → In Progress → Completed

Partially Planned means some portions of the order have been scheduled onto a trip, but not all. An order can also be Cancelled at any point before execution.

Jobs

Job

The core unit of work

Used in
Planner

If an order answers what, where, and when — a job answers how. A job takes an order (or part of an order) and turns it into something that can be scheduled, accounting for practical constraints like vehicle capacity. Jobs are the core unit of work in Tramm — they must exist regardless of whether orders were used. In operations that do not use orders, jobs are created directly without an order driving them.

Key points

  • One order can produce multiple jobs — for example, if the order requires two vehicles, each vehicle gets its own job. Each job fits within one vehicle's capacity.
  • A job can also combine parts of multiple orders if they share the same delivery site and time window — this consolidates smaller orders for efficiency.
  • A job does not have to be linked to an order — it can exist independently for ad hoc or operational requirements.
  • Jobs carry requirements — capacity (size, weight, volume) and any special vehicle or equipment needs. These are used to match the job to a suitable vehicle when the trip is built.
  • Jobs with physical goods have line items — the individual products and quantities being delivered or collected. See the Line Items section below.
  • For crossdock operations, a job can involve an intermediate transfer of goods between two depots before final delivery.
States New → Planned → In Progress → Completed

A job moves to Planned when it is added to a trip. It can also be Cancelled.

Trips

Trip

The vehicle's route for the day

Used in
PlannerTramm MobileFleetvision

A trip is a sequence of planned site visits (stops) assigned to a vehicle. It is created when the scheduler groups jobs together and builds a route. A trip has a start site, a planned start time, and a list of stops in order.

Key points

  • Every trip begins from a start site — the site the vehicle departs from, typically where goods are loaded onto the vehicle. Returning to the start site at the end of the route is optional and is configured on the master route.
  • A trip can be allocated to a specific vehicle, a virtual vehicle (a placeholder when the exact vehicle is not yet confirmed), or left unallocated if imported from an external system.
  • The planned start time is usually set upfront — either from a master route schedule or entered manually by the planner. It is then adjusted if stop time windows require an earlier departure.
  • A trip's combined load requirements — capacity, vehicle class, special equipment — must match the allocated vehicle. The scheduler validates this automatically.
  • Some operations use master routes — recurring trip templates that run on a set schedule (e.g., every Monday). A master route generates actual trips each time it runs.
  • Once a trip is underway, a driver can add unplanned stops via Tramm Mobile — for example, if a new delivery is requested while already on the road.
States Created → Allocated → Pre-Trip → In Progress → Completed

Pre-Trip is when the driver has started the Start of Day workflow at the depot but has not yet set off. A trip can also be Cancelled.

Simple Trip

One vehicle, one client stop, one job.

Multi-Stop Trip

One vehicle, multiple jobs on one route.

Split Order

One order fulfilled by two vehicles running concurrently.

Stops & Activities

Stops and activities are the execution layer of a trip — where the physical work happens.

Stop

Where the work happens

Used in
PlannerTramm Mobile

A stop is a planned visit to a site as part of a trip. Each stop records the planned arrival time, any wait time if the vehicle arrives before the site opens, travel distance and duration from the previous stop, and the list of activities to be carried out there. A stop can contain multiple activities — for example, a delivery and a collection at the same site.

Key points

  • Stops have a time window — the period within which the vehicle must arrive. Arriving too early may mean waiting; arriving too late is a service failure.
  • Sites can also have operating hours — a recurring weekly schedule (e.g., Monday–Friday 07:00–17:00). The system will never schedule a stop outside a site's operating hours.
  • Stops can be added or removed from a trip by the planner. The system recalculates arrival times automatically after any change.
  • Adjacent stops at the same site can be merged into one stop with multiple activities.

How stop duration is made up

What each segment means

  • Arrival Time is when the vehicle reaches the geographic area of the site — not when work starts. If the vehicle arrives before the site opens, it waits. The Wait segment covers that gap.
  • Entry duration is the time it takes to physically enter the site after it opens — gate checks, parking, reaching the loading bay. It is configured on the site in Data Manager. Exit duration works the same way on departure.
  • A stop can have no activities, one activity, or multiple activities. Entry and exit duration pad the total stop time regardless of how many activities there are.
  • If a stop has no activities, it can still have a fixed duration — for example, a driver rest stop with a defined break length.
  • Activity duration is normally calculated from loading rates configured on the job and the site. If the job has an explicit duration set, that value is used instead.
  • Stop duration takes precedence over the sum of its activities. The trip clock advances by the stop's planned duration, not by the total of individual activity times. If activities would take longer than the planned stop duration, they can overflow beyond it — but this is not tracked as a separate status in the system.
Driver states Planned → On the Way → Arrived → In Progress → Completed

These states are driven by the driver's actions in Tramm Mobile. Each transition records an actual timestamp.

Activity

The work done at a stop

Used in
Tramm MobileExecution Manager

An activity is the fulfillment of a job at a stop. It is where the actual loading, offloading, or service takes place. Every delivery or collection job produces two activities — one at each end of the journey.

Activity pairing

  • A delivery job produces a Load activity at the depot (goods onto the vehicle) and a Delivery activity at the client site (goods off the vehicle).
  • A collection job produces a Collection activity at the client site (goods onto the vehicle) and an Offload activity at the depot (goods off the vehicle).
  • A service job produces one Service activity at the site — no goods movement.
  • A Break activity represents a scheduled driver rest stop. No goods are moved; it is time-allocated only.

Activity types

At depot
Load
Goods are loaded onto the vehicle at the depot before the route begins.
At client site
Delivery
Goods are offloaded from the vehicle at the delivery site.
At client site
Collection
Goods are collected from the site and loaded onto the vehicle.
At depot
Offload
Collected goods are offloaded from the vehicle at the depot.
At site
Service
Work is performed at a site — no goods movement involved.
En route
Break
A scheduled driver rest stop. Time-allocated; no loading or offloading.

Line Items

Line Item

The individual goods within a job

Used in
Order ManagerTramm Mobile

Line items are the individual products within a delivery or collection job. Where a job says "deliver goods to Site X," the line items say exactly what those goods are and in what quantities. Line items enable item-level tracking from the moment goods are loaded onto the vehicle to the moment they are confirmed at the delivery site.

What a line item tracks

  • Product — What the item is (e.g., "Long-Life Milk 1L").
  • Ordered quantity — What the client requested.
  • Loaded quantity — What was actually put on the vehicle, confirmed by the driver during Start of Day.
  • Delivered quantity — What the driver actually handed over, recorded at the delivery stop.
  • Returned quantity — Items that came back — refused deliveries, damaged goods, or short deliveries.

Transport Units

  • Some operations group line items into Transport Units — physical containers such as boxes, crates, pallets, buckets, kegs, or drums. A Transport Unit is a named container that holds one or more products.
  • When Transport Units are in use, line items can have sub-line-items — each unit tracks its own product quantities separately from the top-level job quantities.
  • Transport Units are optional. Many operations work with simple line items and no containers. The choice depends on how granularly goods need to be tracked during loading and delivery.

Why line items matter

  • They provide item-level proof of delivery — not just "delivery done" but "exactly X units delivered of Y product."
  • Discrepancies between ordered, loaded, and delivered quantities are captured automatically and can trigger a claim or exception.
  • Barcodes can be configured on line items, allowing the driver to scan each item using their phone's camera rather than entering quantities manually — no dedicated scanning hardware required.
  • Service jobs and break activities have no line items — line items only apply where physical goods are being moved.

On the Road — Tramm Mobile

Once a trip is allocated, it moves from the planning world into the execution world. The driver receives the trip on Tramm Mobile and follows it from the first stop to the last.

Tramm Mobile

What happens during execution

Tramm Mobile guides the driver through each stop in sequence. At every stop, the driver marks their status (on the way, arrived, started, done), carries out the configured workflow steps, and records actual times and quantities. This data syncs back to the system in real time — or queues locally if there is no internet or mobile signal, and syncs as soon as a connection is restored.

Actuals — what the driver records at each stop

Actual What it records
Travel Start Time When the driver marks "on my way" — the vehicle has left the previous stop and is heading to this one.
Arrival Time When the driver marks they have arrived at the stop.
Start Time When the driver starts work at the stop (begins the delivery or service activities).
End Time When all activities at the stop are complete.
Departure Time When the driver leaves the stop and sets off to the next one.
Odometer The vehicle's odometer reading at the stop — recorded when configured per stop type.

These actuals are compared against planned times by Fleetvision to measure on-time performance.

Workflows and ePOD

At each stop, the driver follows a configured workflow — a sequence of steps that must be completed before moving to the next stop. Workflows are defined per stop type and can include any combination of the following:

Customer signature Photos Barcode scanning Quantity entry Text notes Checklist items Odometer reading NFC scan Driver's licence scan

The completed workflow becomes the electronic Proof of Delivery (ePOD) — a digital record that replaces the paper delivery note. The ePOD is linked to the activity and stored against the job.

Depot stops and ad hoc stops

Beyond the planned delivery and collection stops, Tramm Mobile also manages the start and end of the driver's day, as well as any unplanned stops that arise.

Start of Day SOD — The workflow at the depot before the driver sets off. Typically includes a vehicle inspection checklist, an opening odometer reading, and confirmation that goods have been loaded onto the vehicle. Actual delivered quantities are confirmed later, at each delivery stop.
End of Day EOD — The workflow when the driver returns to the depot at the end of the route. Typically includes a final odometer reading and confirmation of any returned goods.
Unplanned Stop Unplanned stop — A stop added to the trip while the driver is already on the road. The driver adds it in Tramm Mobile; it is sent to the system and becomes visible in both Tramm TMS and Fleetvision immediately.

Live Fleet Visibility — Fleetvision

While the driver executes the trip on Tramm Mobile, the operations team monitors everything in real time on Fleetvision.

Fleetvision

What the operations team sees

Fleetvision receives a continuous stream of GPS positions from Tramm Mobile and — where fitted — from vehicle telematics units. It uses this data to track each vehicle's position and progress, compare actual performance against the plan, and generate alerts when something goes wrong.

What Fleetvision tracks

  • Live vehicle positions — Every active vehicle on a map, updated in real time with movement status (moving, stationary, stopped).
  • Trip progress — Which stops have been completed, which are next, and whether the vehicle is running on time, early, or late.
  • Stop matching — Fleetvision automatically detects when a vehicle arrives at a planned stop, even if the driver has not yet marked it in Tramm Mobile. It compares GPS position against the site's location.
  • Planned vs. actual — Scheduled arrival times vs. when the vehicle actually arrived. Lateness is calculated automatically and escalated if unacknowledged.
  • ETAs — Predicted arrival times for upcoming stops, updated as the trip progresses.

Alerts

Fleetvision generates alerts automatically when it detects a deviation from the plan. Unacknowledged alerts escalate to supervisors.

Late Arrival
The vehicle has not arrived at a planned stop within the scheduled time window.
Off-Route
The vehicle has deviated significantly from the planned route between stops.
Unplanned Stop
The vehicle has stopped at a location not included in the trip plan, for longer than the configured threshold.
Overspeeding
The vehicle is travelling above the configured speed limit for the route or zone.
Harsh Driving
Sudden braking, sharp acceleration, or a panic button event, detected via the vehicle's telematics unit.

What the operations team can do

  • Acknowledge and escalate alerts; add comments for record-keeping.
  • Communicate with drivers directly via WhatsApp integration.
  • Allocate or reallocate a vehicle to a trip — including drag-and-drop bulk allocation.
  • Perform a vehicle swap if a vehicle breaks down — reassign the trip to a replacement vehicle mid-route.
  • Force-complete a stop or trip when needed, with a reason captured for the audit log.
  • Review historical trip performance — actual vs. planned timings, stop dwell times, route deviations.

Putting It All Together

A practical example showing how a single order flows from request all the way through to confirmed delivery — across all three products.

Scenario

Pick n Pay Rosebank orders 500 boxes of long-life milk to be delivered from JHB Depot on Tuesday morning between 08:00 and 12:00. The driver is Bob. The operations supervisor is on Fleetvision.

Order An order is created in Order Manager: deliver 500 boxes of long-life milk to PnP Rosebank from JHB Depot, time window Tuesday 08:00–12:00. The auxiliary site is JHB Depot — where goods will be loaded. Order → New
Job The planner (the person using the Planner app) reviews the order. 500 boxes fit in one 4-ton vehicle, so one job is created with two line items: 300 × Long-Life Milk 1L and 200 × Long-Life Milk 2L. Job requirements: 4-ton vehicle, refrigerated. Job → New
Trip The planner creates a trip for vehicle BC12DF GP: start at JHB Depot at 07:30, visit PnP Rosebank, return to depot. Bob is allocated as the driver. The trip is sent to Tramm Mobile. Trip → Allocated · Job → Planned · Order → Planned
Stops The trip has three stops: JHB Depot (Start of Day + loading), PnP Rosebank (delivery), and JHB Depot again (return + End of Day). Each stop sits in Pending until Bob starts working through them.
Activities Two activities are planned: a Load at JHB Depot (500 boxes onto BC12DF GP) and a Delivery at PnP Rosebank (500 boxes to be confirmed on arrival).
Mobile — SOD Bob opens Tramm Mobile at 07:15 and taps Start Trip. He completes the Start of Day workflow — vehicle inspection checklist and confirmation that goods are loaded. At 07:30 he taps Depart. Trip → Pre-Trip then In Progress · Stop (JHB Depot) → Completed
Mobile — Delivery Bob arrives at PnP Rosebank at 08:45 and taps Arrived — Arrival Time is recorded. He taps Start Work and completes the delivery workflow: scans each product barcode using his phone, enters 300 × Milk 1L and 200 × Milk 2L as delivered, and collects the store manager's signature. He taps Done. Stop → Completed · Activity → Completed · ePOD generated
Fleetvision Throughout the trip the supervisor on Fleetvision can see BC12DF GP moving on the live map. When Bob tapped "Arrived" at PnP Rosebank at 08:45, Fleetvision showed the stop as On Time (window closes at 12:00). No alerts were triggered.
Completed Bob returns to JHB Depot and completes the End of Day workflow — records the closing odometer and confirms any returned goods. All stops are done. Trip → Completed · Job → Completed · Order → Completed. The ePOD — with signature, quantities, and timestamps — is stored against the job.

Sites & Locations

Tramm uses two related but distinct concepts — Location and Site — that are easy to confuse. Understanding the difference helps explain how travel times, entry durations, and operating hours are configured.

Data Manager

Location vs Site

What this means in practice

  • When you create a Site in Data Manager, you link it to a Location (or create the Location at the same time). The Location stores the address; the Site stores the operational details.
  • Travel time calculations use the Location's coordinates. Entry and exit duration calculations use the Site's settings. Both play a role in planning.
  • Users work with Sites — you assign jobs to sites, plan trips to sites, and configure time windows on sites. The underlying Location is managed in the background.
  • Fleetvision uses Location coordinates to detect when a vehicle has physically arrived at an address, even before the driver marks arrival in Tramm Mobile.

Terminology

Tramm uses specific terms consistently throughout all three products. These definitions apply everywhere.
